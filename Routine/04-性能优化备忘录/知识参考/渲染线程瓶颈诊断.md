## 一、UE 的三线程模型（2026 还是这套）[D]

> **来源**：
> - UE 官方 "Gameplay Framework" 文档
> - 可可西博客 "UE4 之 Game、Render、RHI 多线程架构"（`https://www.cnblogs.com/coco/...`）— 引用了 UE 源码 `LaunchEngineLoop.cpp` 和 `RenderingThread.cpp`

```
┌──────────────────────────────────────────────────────┐
│  GameThread (GT)  →  业务逻辑：Tick、Actor、动画、物理、蓝图    │
└──────────────────────────────────────────────────────┘
         │ ENQUEUE_RENDER_COMMAND
         ▼
┌──────────────────────────────────────────────────────┐
│  RenderThread (RT)  →  生成平台无关的 RHI Command List        │
└──────────────────────────────────────────────────────┘
         │ ImmediateFlush (DispatchToRHIThread)
         ▼
┌──────────────────────────────────────────────────────┐
│  RHIThread (可选)  →  翻译成 DX12/Vulkan/Metal 调用           │
└──────────────────────────────────────────────────────┘
         │ 提交 GPU
         ▼
┌──────────────────────────────────────────────────────┐
│  GPU                                                  │
└──────────────────────────────────────────────────────┘
```

### 关键事实 [D]

- [D] **GT 跑 N 帧，RT 跑 N 或 N-1 帧，RHI 跑 N 或 N-1 帧**（允许落后一帧，最大并行）— 来源：可可西博客
- [D] **GT 不能领先 RT 超过 1 帧**（否则 `FrameEndSync` 会卡住 GT）— 来源：可可西博客
- [D] **RT 每一帧都 `ImmediateFlush` 等 RHI**（deferred shading 管线），所以 RT 等 RHI = 卡 — 来源：可可西博客

### 移动端特殊 [D]

- [D] iOS 默认**不**开 RHI 线程 — 来源：可可西博客
- [D] Android 视情况，Adreno / Mali 各有偏好
- [D] 移动端打开 RHI 线程前要 Profile

## 二、5 种"卡顿" + 各自在哪条线程 [H]

> ⚠️ 整张表是 [H] 经验整合。每种卡顿的"具体阈值"需要 Profile 自己的项目。

| 现象 | 线程 | 关键 stat | 典型根因 |
|------|------|----------|---------|
| 整帧卡 1-2 秒 | GT (单帧) | `stat memory` | GC / 同步加载 / Blueprint 重新编译 |
| 单帧 spike 50-100ms | RT 单帧 | `stat slow` | 大场景 streaming 触发 / 距离场构建 |
| 持续高帧 | GT 或 GPU 持续满 | `stat unit` | 见下方决策树 |
| 周期性卡顿 | RT + GT 同步 | `stat unit` 看 RT 等待 GT | TickGroup 分布不均 |
| 越走越卡 | GT 累积 | `stat memory` 涨 | 内存泄漏 / 句柄未释放 |

### 决策树 [U] ⚠️ 我推的整合

```
stat unit 看各线程耗时
    │
    ├─ Game 高？
    │   ├─ 看火焰图 → Tick 哪一组 / 哪个 Actor
    │   │   ├─ TG_PrePhysics 高？→ 物理/AI
    │   │   ├─ TG_DuringPhysics 高？→ 物理求解
    │   │   └─ TG_PostPhysics 高？→ 动画 Tick 后处理
    │   └─ 蓝图 Tick 高？→ 替换为事件驱动
    │
    ├─ Render 高？
    │   ├─ RenderThread 高？→ ENQUEUE_RENDER_COMMAND 排队慢 / 录制慢
    │   └─ RHIThread 高？→ 状态切换 / 大量 DrawCall
    │
    ├─ GPU 高？
    │   └─ profilegpu → Pass 排序 → 见 [[Lumen 性能调优]]
    │
    ├─ Present 高？
    │   └─ VSync 等待 → 是好事
    │
    └─ RHIT 高？
        └─ 渲染线程等 RHI → 瓶颈在 RT 提交，不是 GT
```

> ⚠️ 整张决策树是 [U] 我的推论，**不是 UE 官方建议**。

## 三、4 个最常见的多线程坑

### 坑 1：GT 等待 RT（Frame Pacing 问题）[D]（机制真实）

- [D] **症状**：GT 时间很短，但帧时间还是高 — 来源：UE 官方 Frame Pacing 文档
- [D] **根因**：`FrameEndSync` 在等 RT 完成
- [H] 诊断：找 RT 最长 pass 优化

**反直觉 [H]**：有时 GT 优化得再好都没用，因为 RT 慢 1 帧 GT 就得等。

### 坑 2：主线程做渲染资源创建 [D]（机制真实）

- [D] 同步上传 GPU 资源在 `UStaticMesh::Build` / `UTexture::UpdateResource` 路径上
- [D] 对策：FStreamableManager 异步加载
- [H] **永远不要在 `BeginPlay` 同步加载大资源** — 经验

### 坑 3：蓝图 Tick 堆积 [H]（经验）

- [H] 症状：GT 火焰图里 `ReceiveTick` 出现 N 千次
- [D] 诊断命令：`stat actor`（UE 官方 stat）
- [H] 对策：
  - 默认 `PrimaryActorTick.bCanEverTick = false` [D 这是 UE 官方 API]
  - 高频 Tick 改用 Timer [H 经验]
  - 大量同类 Actor 用 Tick 间隔做分帧 [H 经验]

### 坑 4：物理异步开关不对 [D]（CVar 真实）

- [D] `p.AsyncPhysicsTickEnabled` CVar 存在
- [D] `p.MaxPhysicsSubsteps` 存在
- [H] 症状：物理明显卡 GT，但 TG_DuringPhysics 时间不长 — 经验
- [H] 对策：默认开 async physics — 经验

## 四、4 个被忽视的多线程调优 CVar [D]

> **来源**：UE 官方 CVar 文档

```ini
# 1. 允许 RT 落后 GT 一帧（默认开，但很多人不知道）
r.OneFrameThreadLag = 1

# 2. 物理异步
p.AsyncPhysicsTickEnabled = 1

# 3. RHI 线程（PC / 主机推荐开）
r.RHICmdBypass = 0
r.RHIThread = 1

# 4. 任务图并行
taskgraph.parallel.for
```

- [D] 所有 CVar 名都来自 UE 源码
- [H] "PC/主机推荐开" 是经验

## 六、移动端专属 [D]（基本事实）+ [H]（推论）

- [D] 移动 GPU 是 **TBDR**（Tile-Based Deferred Renderer）— UE 官方文档
- [D] 大三角形 / 大 fill 区域反而比小 DrawCall 更贵 — UE Mobile 文档
- [D] **MSAA 在移动端几乎不要用**（TBDR 本来就是 deferred）— UE Mobile 文档
- [D] 永远检查 `r.Mobile.*` 系列 CVar — UE 官方

- [H] "PC 的'减少 DrawCall'在移动端可能是错的" — 经验
