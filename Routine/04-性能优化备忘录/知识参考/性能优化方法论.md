# 性能优化方法论 — Profile 的正确姿势

> **声明：本文是经验/启发式整理，不是单一来源的复述。每条结论按可靠度分三档标记。**
> - **[D]** Documented — 有官方文档/GDC 演讲/源码可证
> - **[H]** Heuristic — 行业普遍共识或资深团队实践
> - **[U]** Unverified — 我的推断，需要你 Profile 验证

---

## 一句话心法 [H]

**性能优化是"消除假设"的过程，不是"试错"的过程。** 每次优化前必须能回答：现在的瓶颈在哪？为什么？预期能省多少？优化后实际省了多少？

---

## Profile 黄金三问（动手前必答） [H]

1. **我看到的是症状还是根因？**
   - [H] "帧率 30fps"是症状，"GPU base pass 22ms"才是根因方向
2. **这个数字在目标平台上正常吗？**
   - [D] PC 60fps = 16.6ms；移动 30fps = 33ms；主机 60fps = 16.6ms 但通常 80% 预算给 GPU（来源：UE 官方 Scalability 文档）
3. **我准备优化的部分，理论上能省出多少预算？** [H]
   - [H] 如果最多省 0.5ms 而当前差 3ms，就别动这里

## 调试铁律（违反任何一条都意味着在浪费时间）

### 1. 先关掉你怀疑的功能，再开回来 [H]
最便宜的实验就是 A/B。如果你怀疑某材质贵，做一个空材质替进去看帧时间变化。**不要在不理解的情况下改实现**。

### 2. 一次只改一个变量 [H]
- 改材质 + 改光照 + 改 mesh → 帧率涨了 → 哪个起的作用？**不知道 = 白做**
- 永远只动一个东西，Profile 一次，再动下一个

### 3. 在能复现的场景下 Profile [H]
"我们家项目里偶发卡顿"是性能优化最难的问题。先做一个最小复现 demo，把问题稳定下来，再开始分析。
- 跑同一段路径
- 切同一个摄像机角度
- 在同一秒内触发

### 4. Development 构建，不是 Shipping [D]
- [D] Shipping 禁用了 stat、Insights 大部分计数器、断言（来源：UE 官方 Profiling 文档）
- [H] 但 Shipping 的代码路径和 Development 不同（条件编译、Stripped 节点、不同的 cooking）
- **判断优化效果在哪种构建下都 Profile 一次**

## 性能瓶颈识别 — 心智模型

### 三线程瓶颈来源 [D]

**UE 引擎一帧的时间分布在三条主要执行流上：**（来源：UE 官方 "Gameplay Framework" 文档，UE 5.7 链接：`https://docs.unrealengine.com/5.7/en-US/gameplay-framework`）

```
┌─────────────────────────────────────────────────────────┐
│  GameThread  │ RHIThread  │  GPU  │  Present  │ VSync   │
│  Tick逻辑    │ 提交命令   │ 渲染  │ 缓冲交换  │ 等待    │
└─────────────────────────────────────────────────────────┘
        ↓             ↓           ↓
   stat game    stat rhi    stat gpu / RenderDoc
```

- [D] `stat unit` 显示的列：Frame / Game / Draw / GPU / RHI / RHIT / Present — 这是 UE 内置 stat 的标准输出
- [H] 哪个线程耗时最高决定了你下一步往哪查
- [H] 决策树下面有，是我的经验整合，**不是官方建议**

### 诊断流程 [H] ⚠️ 这是我推的决策树

```
帧时间 > 目标
    │
    ├─ 看 stat unit：Frame / Game / Draw / GPU / RHI / RHIT / Present 各占多少
    │
    ├─ GPU 占比最大？
    │   ├─ 是 → RenderDoc 单帧捕获，profilegpu 命令细分到 RenderPass
    │   │     ├─ BasePass 高 → 看 GBuffer 复杂度、材质复杂度
    │   │     ├─ Lumen 占比 → 看屏幕探针 / 表面缓存 / 光线长度
    │   │     ├─ Nanite 占比 → 看几何密度、视锥/遮挡剔除
    │   │     └─ PostProcess → 屏幕空间 Pass 链
    │   │
    │   └─ 否 → CPU 侧
    │       ├─ GameThread 高 → Tick、动画、物理、蓝图
    │       ├─ RHI 高 → 状态切换、Shader 切换、Buffer 上传
    │       └─ RenderThread 高 → 命令录制，Mesh Draw Command 合并
    │
    └─ Present 等待 / VSync 高？
        └─ GPU 已经做完，等显示器 → 这是好事，瓶颈在别处
```

> ⚠️ **承认：** 这个决策树是 [U] 我推的整合。具体 Pass 名（BasePass、Lumen、Nanite）来自 UE 5.0+ 实际 Pass 列表，但"占比 → 查哪"的映射是经验。

## 几个反直觉的真相

### "优化最贵的 1%" 能吃掉 80% 时间 [H]
Pareto 分布。一次 Profile 后，先找 top-3 耗时函数。优化它们。不要去碰那些"感觉上不对"的地方。

### 优化 = 增加可选项，不是减少工作量 [H]
高手不会问"我能不能不画这个物体"，他们会问：
- "这个物体能不能用 LOD"
- "能不能每 3 帧更新一次"
- "能不能在另一个线程上算"
- "能不能合并到另一个绘制里"

### 移动端 ≠ PC 的"慢版" [D]
- [D] 移动 GPU 是 **TBDR**（Tile-Based Deferred Renderer），不同渲染路径（来源：UE 官方 Mobile Rendering 文档）
- [D] 移动 CPU 是 **big.LITTLE**，线程亲和性重要
- [H] 移动端带宽更紧，但 fill rate 不一定差
- [H] 一个 PC 上的优化（如合批）可能对移动端完全无效

### 测量永远不嫌多 [H]
- 一个判断至少 3 个数据点
- 同一个场景跑 10 帧取中位数（首尾的 GC / 流送 spike 不可信）
- 多次开关不同功能做 A/B

## 经验沉淀 — 心智速查 [H]

> ⚠️ 整张表是 [H] 经验整合。每个症状的具体阈值/数值需要你 Profile 自己的项目才能定。

| 卡顿形态 | 第一时间查 | 标志性信号 |
|---------|-----------|-----------|
| 整帧卡 1-2 秒 | GC / 流送 / 同步加载 | `stat memory` 看增量 |
| 单帧 spike 50-100ms | 异步加载 | 包追踪里的 `LoadPackage` |
| 持续高帧时间 | 主线程或 GPU 持续满载 | `stat unit` 看分布 |
| 周期性卡顿（每 30 帧一次） | Tick 间隔 / 定时器 | `stat slow` 看 actor 列表 |
| 越走越卡（>10s 级别） | 内存泄漏 | `stat memory` 总值在涨 |
| 切场景后第一帧卡 | Shader 编译 / Streaming | 第一次该材质首次出现 |
