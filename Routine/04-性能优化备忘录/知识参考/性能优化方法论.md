# 性能优化方法论 — UE 官方说法 + 普适原则

> 本笔记**只收录**两件事：

> 2. 普适的工程原则（不是性能数字）

## 一、UE 官方对 Profile 流程的明确说法 [D]

> **来源**：UE 官方 [Unreal Insights Overview](https://docs.unrealengine.com/5.7/en-US/unreal-insights-overview-in-unreal-engine)

### 1.1 官方推荐的工作流

> "现场调试"：**`stat unit` / `stat game` / `stat gpu` / `stat unitgraph`**（实时看）
> "性能调优"：**Unreal Insights**（事后分析 utrace）
> "像素级问题"：**RenderDoc** 单帧捕获

### 1.2 官方文档给出的"为什么用 Insights" 论据 [D]

> - 任意线程任意时段的**完整调用栈**、内存变化、IO、Bookmarks、Counter
> - 跨帧对比（这是 `stat unit` / `profilegpu` 做不到的）

---

## 二、UE 官方对 Shipping vs Development 构建的明确说法 [D]

> **来源**：UE 官方 [Profiling and Optimization](https://docs.unrealengine.com/5.7/en-US/profiling-and-optimization-in-unreal-engine) 文档

- **Shipping 构建**：禁用 stat、Insights 大部分计数器、断言
- **Development 构建**：保留所有 Profile 工具

→ **判断优化效果在两种构建下都 Profile 一次**（这是 UE 官方的明确建议）。

---

## 三、UE 官方对移动端 ≠ PC 的明确说法 [D]

> **来源**：UE 官方 [Mobile Rendering](https://docs.unrealengine.com/5.7/en-US/mobile-renderer-in-unreal-engine) 文档

- 移动 GPU 是 **TBDR**（Tile-Based Deferred Renderer）—— 渲染路径与桌面 GPU 不同
- 移动 CPU 是 **big.LITTLE** —— 线程亲和性重要
- 移动端 **MSAA 几乎不要用**
- 永远检查 `r.Mobile.*` 系列 CVar

---

## 四、UE 官方对 Lumen / Nanite 目标平台的明确说法 [D]

> **来源**：UE 官方 [Lumen Technical Details](https://docs.unrealengine.com/5.7/en-US/lumen-technical-details-in-unreal-engine)

- Lumen / Nanite 的**目标平台**：next-generation consoles（PS5 / Xbox Series）+ high-end PCs
- Lumen **不支持**：旧主机（PS4 / Xbox One）、VR
- Lumen 在次世代主机的 **Epic scalability**：GI + 反射 ≈ **8 ms**（1080p internal，靠 TSR 推到近 4K）

→ 如果你的目标平台不在这些范围内，**Lumen / Nanite 可能根本不适合**，先确认平台。

---

## 五、UE 官方对 Lyra 性能目标的明确说法 [D]

> **来源**：UE 官方 [Lyra Sample Game](https://docs.unrealengine.com/5.7/en-US/lyra-sample-game-in-unreal-engine) 文档

- Lyra 是 **"Learning resource"**（学习资源），不是商业游戏模板
- 它展示了 UE5 框架在 sample 规模下的最佳实践

→ 不要直接把 Lyra 的数字当作商业项目的目标。

## 六、普适工程原则（不是性能数字）[H]

> 以下是**普适的工程原则**，不是性能数字。任何具体阈值 / 收益数字本文**不主张**。

### 6.1 Profile 黄金三问（行动前必答）[H]

1. **我看到的是症状还是根因？**
2. **这个数字在目标平台上正常吗？**（参考 UE 官方 Scalability 文档的目标）
3. **我准备优化的部分，理论上能省出多少预算？**——参考 `[[Unreal Insights 帧分析实战]]` 录数据

### 6.2 调试铁律 [H]

1. **A/B 铁律**：先关掉你怀疑的功能，再开回来。**不要在不理解的情况下改实现**。
2. **一次只改一个变量**：同时改 N 个东西 → 涨了 → 哪个起的作用？**不知道 = 白做**。
3. **先在最小复现下 Profile**：偶发卡顿先做最小 demo，再分析。

### 6.3 反直觉（普适原则）[H]

- **优化最贵的 1% 能吃掉 80% 时间**（Pareto 分布）：先找 top-3 耗时函数
- **优化 = 增加可选项，不是减少工作量**：高手问"这个能不能 LOD / 每 3 帧更新 / 另一个线程 / 合并绘制"，而不是"能不能不画"
