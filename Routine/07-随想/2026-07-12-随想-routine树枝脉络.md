# Routine 里的"主枝叶":一层能动的结构

> **一句话:** Routine 不是一个文件夹,是一棵活着的树。**主干**告诉你根往哪扎,**枝干**告诉你怎么向光长,**树叶**让你活过当季。**不区分这三层,任何努力都会在"焦虑中全做"里被稀释。**

---

## 0. 为什么要写这篇

Routine 摆在我面前已经 6 周。从最初"按周一到周日跑 Lumen 学习" → "AI 工具链方法论" → "day-job RAG + Mac Game Harness" → "P0 10 项 + 季度复盘"——这条线**至少换过 3 次主语**。

每次回头看,都发现同一个症状:**重要的事情越做越多,但"什么算重要"的尺度一直在漂移**。

01-AI-GameDev-Resource-Index 列了 8 个主题、5 条证据、1 个本周动作——**可执行、可对照**,但它是"叶子级"。
05-技术雷达 P0 列了 7 个工具 + 3 个渲染特性——**有时效、有回顾日**,但它是"枝干级"。
06-职业复盘每次都说"主线 = day-job 喂 LLM"——**这是主干级**,但从来没写在显眼位置。
90-milestones(我还没看)和 00-AI-Agent-Routine-Overview 里那句"Agent-First / Routine-First / Evidence-Based / Composable"——**是主干的元规则**。

四层文件都在,但**谁压谁、谁给谁让路**没有显式约束。所以主干会被"今天的热点"挤变形,枝干会因为"完成 9 周 shader"自我庆祝,树叶会反过来冒充枝干。**Routine 因此变得勤奋但失重。**

这篇随想把这棵树显式画出来,留下可执行的判断标准。

## 1. 三层的定义(我给 Routine 用的版本)

### 1.1 主干(Trunk)— 不可动摇的方向锚点

> **判定问题:** 3 年后这个方向还在不在?
> **数量上限:** 3-5 个,**不超过 5**。
> **时间尺度:** 3-5 年不换。

Routine 当前的主干,在我观察里是这 3 件事:

1. **day-job 真相:RAG + Mac Game Harness,目标"喂 LLM 用 UE 特性"**——这是 06 复盘和 05 雷达反复重申的"主线 C"。错位最严重的就是它——常被淹没在"今天该读哪篇论文"的执行层里。
2. **UE5 引擎本身**——Lumen / Nanite / VSM 不是三个独立主题,它们是"引擎对 RAG 必须暴露的能力面"。05 雷达 2026-07-12 的"P0 渲染特性补丁"就是把这件事从"枝干"拉回"主干分支"。
3. **AI 工具链标准**——MCP / Claude Code / NNE 不只是工具,它们是"LLM 时代引擎程序员的工作姿势"。一旦姿势换,以前所有积累都要重新接枝。

主干的特征是:**一旦换了,以前的工作大部分变成"历史资产"而不是"现役能力"**。所以主干要**少、要硬、要能抗住热点诱惑**。

### 1.2 枝干(Branch)— 持续 6-18 个月的方向延展

> **判定问题:** 这个方向 6-18 个月后还会推进吗?它在为主干解决哪个具体缺口?
> **数量上限:** 5-8 个,**不与主干重复**。
> **时间尺度:** 半年到一年半。

Routine 当前的枝干(从 01/03/05/04/02 抽出来):

| 枝干 | 当前形态 | 服务的主干缺口 |
|------|---------|---------------|
| **神经 Shader 9 周方向**(W1-W9) | 03-Shader 整库,9 篇完成 | 神经材质 / BRDF / DLSS 风格降噪 → day-job 训练数据 |
| **P0 工具链 7 项** | 05 雷达 P0 上半区 | 让 LLM 能"读 + 改 + 验证" UE 工程 |
| **P0 渲染特性 3 项**(Lumen/Nanite/VSM) | 05 雷达 P0 下半区 + 03 W3-W6 + 04 性能 | 让 LLM 能正确"调用" UE 渲染管线 |
| **引擎源码分析** | 02 引擎源码分析库(5 篇性能瓶颈 + W27 Nanite 起头) | 给 RAG 提供"内部机制"深度证据 |
| **Mac 平台 vault** | 2026-07-12 才决定要建,完全空白 | 喂 LLM 的运行时约束(不是"工作姿势"了) |
| **AI-Tasks 主题轮转** | Lumen / Nanite / VGM / Chaos / Water / UnrealMCP / NeuralRendering / PCG / MassAI | 主题化学习的"容器" |

枝干的特征:**它是主干的"工作分解",但有独立寿命**。一个枝干可以在服务完一个主干缺口后被修剪,但其方法论沉淀可以升入下一轮主干。神经 Shader W1-W9 就是例子——W7 完成后,DLSS 风格的时域重建会进 day-job 训练数据;W9 NRC 也是。

### 1.3 树叶(Leaf)— 时效性强的具体执行单元

> **判定问题:** 3-12 个月后这件事还重要吗?它给哪根枝干供能?
> **数量上限:** **不设上限,但每片叶子的存活期必须能数出来**。
> **时间尺度:** 1 周 - 12 个月。

Routine 当前的树叶举例:

- W28 这次复盘(2026-07-12) / W27 复盘(7-05) / W26 复盘——**日记/复盘类叶子**
- 单篇 shader 笔记(W3 Lumen 反射降级, W4 Lumen GI 漫反射, W5 Nanite 材质管线, W6 VSM)——**技能落地叶子**
- 一次 MCP server 配置(2026-07-09 试通的 filesystem server)——**工具链落地叶子**
- 一篇性能瓶颈记录(VSM 页溢出 / Lumen 反射开销)——**Profile 叶子**
- 一篇职业复盘(2026-07-05 那次"是否启动 AI 训练 pipeline UE5 直通车项目化"决策)——**决策叶子**

树叶的特征:**它是 Routine 的"呼吸",但不是 Routine 的"骨架"**。一片叶子落了就落了——但如果你发现一整根枝干上全是"枯叶"或者"一片叶子都没长",那这根枝干本身就要被诊断。

## 2. 如何把握——三个判断标准

### 2.1 主干的方向性:**用"3 年后还不在做"反向锁定**

> **标准:** 写下一件你现在最想做、最焦虑的事,然后问自己:"3 年后我还会做它吗?"
> - **是** → 它是主干的延伸,值得持续投入。
> - **否** → 它是当前主干的"应急叶子"或"过时枝干"——保留执行,但别让它指挥下一周的工作。

Routine 现状里**最容易冒充主干的是 P0 工具链 7 项**。它们每一项都很"新、很热、很对",但**它们不是主干**——它们是枝干。具体说:

- DLSS / FSR 是 5 年内大概率长期存在的工具,**但**它服务于"UE 渲染管线的画面生成"这个主干,本身不是主干。
- UnrealMCP / Claude Code 是 2-3 年的工作姿势,**但**它服务于"LLM 时代怎么编程"这个主干,本身不是主干。
- Lumen / Nanite / VSM 是 UE5 引擎当前的能力面,**但**它们服务于"LLM 能否正确调用 UE 渲染管线"这个主干,本身不是主干。

**真正的主干是那 3 件"3 年后还在不在"的事**:day-job 真相、UE5 引擎能力面、LLM 时代工作姿势。

这就是为什么 05 雷达 2026-07-12 把"工具链 7 项"和"渲染特性 3 项"分两个子节:不是简单分桶,是承认它们服务的是**不同的主干**。但**雷达本身不应该有 10 个 P0**——P0 数量太多 = 主干边界模糊的信号。真正的对策是:**P0 数量应该限制在 5-7 项**,超出就降级为 P1,或者承认"其中有几项是 P0 候选,季度末再确认"。

**主干方向性的可操作标准**:每季度第一周,写一段不超过 200 字的"主干自检"——"3 年后我还在做什么?这 3 件事之间什么关系?"——写不出来,说明主干在漂移。

### 2.2 枝干的敏锐性:**用"对主干的瓶颈侧"反向校准**

> **标准:** 给每根枝干一句话定位:"它解决主干的什么瓶颈?"说不出来的枝干,该修剪。

Routine 里**最容易模糊主枝关系的是"AI-Tasks 主题轮转"**。Lumen / Nanite / VGM / Chaos / Water / UnrealMCP / NeuralRendering / PCG / MassAI / LearningAgents 9 个主题——它们每一个都是"主题",但**不是每一根都是"枝干"**。

- Lumen / Nanite / VSM → 是 05 雷达 P0 渲染特性轴的**枝干**(已在 07-12 校准)
- UnrealMCP / NeuralRendering / PCG / MassAI / LearningAgents → 哪些是"枝干",哪些是"工具链轴的 P0",要分开评估
- VGM / Chaos / Water / VSM → 这四个里,VSM 升入 P0 渲染特性轴了,其它三个是不是"已归档"或"季度末评估"?

**枝干敏锐性的可操作标准**:

1. **每根枝干写一句"它为主干解决了什么"**——3 个月回看,做到了就保留枝干身份;做不到就降级为"项目/工具"或归档。
2. **每根枝干必须容纳"变化"**——3 个月内技术换了,枝干还在(方法论可复用),那是健康枝干;枝干随技术死掉,说明它是伪装成枝干的树叶。
3. **每根枝干必须有"修剪记录"**——W7 DLSS 完成 ≠ 整个神经 shader 方向完成,要看 W10 是什么、是不是该升格为"神经渲染 vs 传统渲染"的方法论对比,而不是继续每周加 W10/W11/W12。

神经 Shader W1-W9 的现状是"9 周做满、9 篇质量自评优"——**但枝干本身没有升级**。这就是"叶子做得好,但枝干没长粗"的典型现象。建议:09 月底,把这 9 周做一次"枝干重整":升 3-4 篇进入 day-job 训练数据模板,归档 2-3 篇只作为对照,留 1-2 篇作为"持续维护"。

### 2.3 树叶的时效性:**用"它能活多久 + 它给谁供能"双向过滤**

> **标准:** 每片叶子必须有"出生日期 + 预计退役日期 + 供能的枝干/主干名"。

Routine 现状里**最容易失控的是 P0 工具链 7 项的"叶子"**——每项都有 "下次回顾 2026-07-10" 的截止日,但**没有"供能谁"的显式记录**。这就导致:

- DLSS 4.5 复评日到了 → 复评 → 写 1 条 04 性能优化 → 完。但**这条叶子未来是给"画面生成枝干"还是"day-job 训练数据枝干"供能?没说。**
- NVIDIA-ACE 复评日到了 → 复评 → 写 1 条 06 复盘 → 完。但**ACE 是为了"AI 队友原型"还是"AI-NPC 平台"枝干供能?没区别。**

**树叶时效性的可操作标准**:

1. **每片叶子(每条 06 复盘 / 每篇 shader 笔记 / 每条 04 性能记录)在 frontmatter 里加 3 个字段:**
   - `born: YYYY-MM-DD` — 出生日
   - `expires: YYYY-MM-DD` — 预计退役日(默认 6 个月)
   - `feeds: [[主干名]] 或 [[枝干名]]` — 供能对象
2. **季度末统一过一遍"过期叶子"**:过期了但还在用 → 升格为"持续维护";过期了且没在用 → 归档到 `99-归档/`;还没到期 → 留着。
3. **每片叶子的"出生"必须能回答 1 个问题:它在为哪个具体决策/瓶颈/读者供能?** 回答不了的叶子 = 自嗨叶子,该砍。

## 5. 三条现在能落地的动作

写完理论,留 3 件**本周内**能做的事:

1. **给 Routine 里每篇 .md 补 frontmatter**(工作量中等):
   - `born: YYYY-MM-DD`(创建日)
   - `expires: YYYY-MM-DD`(默认 +6 个月,日记/复盘 +3 个月,P0 工具链 +12 个月)
   - `feeds: [[目标枝干/主干]]`(给谁供能)
   - 第一周先做 06 复盘日志和 04 性能优化备忘录——这两个是"叶子最密集、错位最严重"的区。

2. **修剪 01-AI-GameDev 资源索引**:把那 304 行拆出 8 个 AI-Tasks 子目录(已经存在 9 个 AI-Tasks 主题,索引是它们的"伪主干")。索引恢复成"5 分钟可读完 + 本周动作"形态。**这件事不做,枝干 G(AI 工具链)的"主题化"永远是空话**。

3. **写第一份"主干自检"**(200 字以内,贴在 00-README 顶部):回答"3 年后我还在做哪 3 件事?"——写不出来就先写"目前认为的 3 件,但还在漂移"。**这是 Routine 的锚**,以前从来没有显式写过,这就是为什么主干会被热点挤变形。

## 6. 最后一句

Routine 不是日程表,是棵**会按你浇水的方向长的树**。

你浇在最显眼的那片叶子上,叶子就疯长、变重、把枝干压弯;
你浇在枝干分叉处,枝干就分叉、变多、把主干撑开;
你浇在主干根部,主干就扎根、变粗、扛得过季节。

Routine 的勤奋,要看**浇在哪**。9 周 shader 全做完 ≠ 神经 Shader 枝干变粗;10 项 P0 雷达 ≠ 工具链枝干方向正确;304 行资源索引 ≠ 知识库结构健康。

**主枝叶区分清楚,Routine 才有"长"的可能。否则只是"忙"。**
