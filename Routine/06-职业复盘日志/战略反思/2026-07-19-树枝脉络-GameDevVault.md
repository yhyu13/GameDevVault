## 一、先画树：什么是什么

### 主干——十年磨一剑，不会弯曲的部分

**主干只有一根：游戏引擎程序员的工程穿透力 + LLM 时代的"RAG 语料生产 + 工具设计"双重身份。**

具体是什么？

- 看到一份 UE5 引擎源码，能立刻定位关键类（`FVirtualShadowMapArrayData` / `Nanite::FCluster` / `FSceneView`），画清楚数据流和控制流
- 看到一段 HLSL / MSLL shader，能判断是否符合渲染管线约束（half / float 精度、采样器绑定、MRT 支持）
- 看到一个 LLM 训练数据样本，能判断是否 grounded、是否有事实性错误、是否覆盖核心 API
- 看到一个 MCP 工具设计，能判断"是否会让 LLM 调用导致编辑器 Fatal error"（W28 Run 9 `set_property` 崩溃就是反例）

这根主干不是这周种下的。它是 Vault 里一页页源码养出来的，是 inspector 里一行行跟出来的。**它不随技术潮流弯曲。** 今年热 MCP，明年热 Agent，后年不知道热什么，但**"读得懂引擎源码、写得动训练数据、设计得了工具接口"**这三件事的组合不变。

> 主干的本质：判断力。判断什么是 day-job 真杠杆，什么是行业噪音，什么只是包装了旧概念的新词。
>
> **与 AIResearchVault 那篇的区别**：那边的主干是"数学严谨性 + 实时系统背景"。这边的不同是**加了 LLM 维度**——day-job 不是纯引擎程序员，是**给 LLM 用的 RAG + Mac Game Harness**。所以穿透力的"穿透"目标从"自己懂"变成"LLM 也能懂"。

### 枝干——正在分叉的新生长方向

**目前有三根主枝干在争夺养分 + 一根刚破土的侧枝：**

| 枝干 | 状态 | 与主干的连接 | day-job 杠杆 |
|------|------|-------------|------------|
| **Lumen / Nanite / VSM 升"已掌握"** | Lumen 最粗（57 KB 源码 + 3 论文 + 2 shader + 2 性能），Nanite/VSM 中（470+340 行源码追踪）| 直接连：源码 → 笔记 → 训练数据 | 渲染三大特性是 LLM 调用 UE 渲染管线的核心知识 |
| **MCP / Tool Use / Agent Harness** | 中：9 phase patch + 15 工具 + smoke test 7 轮 + Run 9 真实 bug 捕获 | 间接连：MCP 工具 → 编辑器状态 → 训练语料 ground truth | 主线 A，day-job 最直接杠杆 |
| **Training Pipeline / 3B 微调 / RAG 索引** | 中：6 阶段流水线 + grounding.json 4113 行 + MCP-grounded v2 | 间接连：训练数据 → 微调模型 → LLM 能力 | 主线 B，day-job 核心交付物 |
| **Mac 平台 (Metal RHI)** | **幼龄**：W29 起头 4.1KB 索引 + 5 anchor + 9 维度差异矩阵 | **几乎没连**：刚破土 | day-job 工程挑战，未来 30 天生死线 |

> 枝干的核心诉求：**敏锐性**。要判断哪根枝干值得继续长、哪根应该剪除、哪根需要嫁接。

### 树叶——当季繁茂、下季脱落的产出

**树叶是 GameDevVault 中时效性最强、更替最快的部分。** 截至 W29：

| 树叶类型 | 数量 | 占比 |
|---------|-----:|-----:|
| 互动式 QA 卡牌 | **27 张**（7 P0 + 3 P1 + 3 P2 + 9 shader + 5 W29）| ⭐⭐⭐ |
| Shader 案例集 (C01-C09) | 9 篇 | ⭐⭐ |
| GDC 2026 paper notes | 8 篇 | ⭐⭐ |
| 每周复盘 + 战略反思 | 3 篇（W27×2 + W28 + W29 + 本篇 = 5）| ⭐ |
| 性能瓶颈案例 | 5 篇 | ⭐ |
| UE5 源码分析（除 VSM/Nanite 追踪外）| ~36 篇 | ⭐⭐ |
| **W29 新增** | **+38.6 KB day-job 语料资产** | — |

树叶的价值在于**当季的光合作用**——它们让 vault 看起来茂盛、让学习过程有可见的进度条、让 day-job RAG 索引可立即拼装。

但树叶的问题在于：**它们会掩盖枝干的真实生长状况。**

W29 产出 14 commits、+38.6 KB 语料资产。**从外部看，这棵树极其茂盛**。但翻开土壤看：

- 渲染三特性中，**只有 Lumen 真正算"厚"**，Nanite/VSM 各只有 300-500 行源码笔记
- Mac 平台只有 4.1KB 索引页，**9 维度差异矩阵里没有一项有实质内容**
- Training Pipeline 的 `outputs/models/` 仍是空目录，**0 个 checkpoint**
- UE5_Extending_MCP 的 `.github/workflows/` 仍未建，**没有 CI**
- 3 份复盘（05 Weekly-Log / 06 详细 / 07 摘要）信息重叠 60%+，**反思层膨胀**

> 树叶的核心法则：**时效性**。当季不摘就会枯黄，但摘太多会耗尽枝干的长高能量。

## 二、问题诊断：养分在哪里错配了

### 错配 1：QA 卡牌抢走了源码追踪的养分

这是 GameDevVault 最严重的错配。

**W28 + W29 累计产出**：

| 类型 | 数量 | 平均行数 | 总行数估算 |
|------|-----:|--------:|----------:|
| QA 卡牌 HTML | 22 + 5 = **27 张** | ~700 行/张 | **~18,900 行** |
| 源码追踪 | 2 篇 (VSM 470 + Nanite 340) | ~400 行/篇 | **~810 行** |

**比例 23 : 1**。每写 23 行 QA 卡牌才写 1 行源码笔记。

为什么？

因为**生产树叶的边际成本远低于培育枝干**。写一张 QA 卡 = `gen_p0_cards.py` 跑一遍 + 填 12 道题。改引擎源码 = 翻 `Engine/Source/Runtime/Renderer/` 几百个文件 + 找到 `FVirtualShadowMapArrayData` 的关键字段 + 画数据流图 + 写 cache 命中分析。

**大脑喜欢树叶**。但**改引擎源码**才是 day-job "LLM-driven UE on Mac" 真正需要的素材——LLM 用 RAG 检索时，需要的是结构化的引擎内部机制描述（"VSM 的页表 LRU 在 `FVirtualShadowMapCache` 中按 clipmap 维护"），而不是 12 道选择题。

### 错配 2：主干的能量没有输送到 Mac 枝干

这根主干最锋利的部分——UE5 源码穿透力——目前主要供给给了**渲染三特性**这根枝干（57 KB Lumen 源码 + 470 VSM + 340 Nanite）。

但 day-job **Mac Game Harness** 需要的是**Metal RHI 穿透力**——这根**幼龄林层**目前只有 4.1KB 索引 + 5 个 anchor，**9 维度差异矩阵里 0 行实质内容**。

W28 复盘自爆："Mac 零笔记"。W29 起头了索引页。但 30 天内如果还是 4.1KB，**Lumen 升"已掌握"的"Mac Metal RHI 上跑通"门槛就过不了**——8/7 月度回顾时会被打回。

**这是一个被忽视的嫁接点**：渲染三特性的源码笔记（如何画 Lumen Surface Cache 数据结构图）→ 同样的方法论可以画 **Metal RHI 的 `MTLBuffer` 内存对齐与 DX12 `D3D12_HEAP_TYPE` 的差异**。这些是主干能量向 Mac 枝干输送的通道，但目前只存在于"待做"列表里。

### 错配 3：三份复盘的资源浪费（"反思层肥胖症"）

W29 复盘同时存在 **3 个版本**：

| 复盘 | 行数 | 角度 |
|------|-----:|------|
| `05-技术雷达/Weekly-Log/W29-2026-07-19/W29-周复盘.md` | ~200 | day-job 视角（OKR 跟踪 + 季度进度）|
| **`06-职业复盘日志/2026-07-19-W29周复盘-...md`（本批写的）** | **~350** | 行为反思（11 段：评分/亮点/卡点/承诺兑现）|
| `07-周报/2026/2026-W29-周报.md`（待写）| ~50 | 4 段式摘要（Input/Output/Connection/承诺兑现率）|

**问题**：
- 05 和 06 信息重叠 60%+（同一批 commits、同一组承诺兑现）
- 写 06 详细版的时候，会下意识去覆盖 05 已经覆盖的内容
- 用户读 3 个文档才能拼出 W29 全貌，**反思的边际效用递减**

> **本篇就是诊断本身**——3 份复盘的资源占用 vs 1 份深度反思 + 1 份摘要的对比，决策应该在 W30 末做出来。

### 错配 4：树叶配额的边界没设

W28 + W29 累计 27 张 QA 卡牌。但**没有配额**——什么时候该停、什么时候该继续、什么时候该"冻结只通过自动化脚本生成"，全凭每天的心情。

参考 AIResearchVault 那篇的建议：

| 树叶类型 | 配额 | 现状 | 偏差 |
|---------|------|------|------|
| QA 卡牌 | 冻结（仅通过 `.tools/gen_*_cards.py` 生成）| 手工 + 脚本混合 | **超量** |
| 复盘日志 | 每周 1 份 | W27×2 + W28 + W29 = 4 份 | 略超 |
| 性能瓶颈案例 | 每周 1-2 条 | W28 0 → W29 3 | 合格 |
| 雷达 P0 复评 | 每月 1 次 | W28 已做 P0 渲染特性补丁 | 合格 |

## 三、如何把握三者的节奏

### 主干的方向性——每年审视一次，不要频繁动摇

主干的方向不是每周甚至每月能变的。它是"我是谁"的答案。

目前的判断：

- **"UE5 引擎程序员 + LLM 训练 / RAG / Agent Harness"** 仍然是最稀缺的组合。游戏圈懂 UE5 的人多，但**懂 UE5 同时能设计 LLM-friendly RAG 语料格式**的人极少。
- **不要放弃渲染三特性**。它是主干的训练场——每篇引擎源码追踪都在锻炼同一种穿透能力。但**这个能力需要从 Lumen/Nanite/VSM 输送到 Mac 平台**。
- **不要盲目追新潮**（Agent / 具身智能 / 世界模型）。这些是**新长出来的树苗**，不是这棵树的主干。

> **主干法则（GameDevVault 版）**：每年确认一次方向。**确认后不因短期潮流摇摆**。W28 立的"day-job = LLM-driven UE on Mac"在 8/7 月度回顾前不重新审视。

### 枝干的敏锐性——每月修剪一次，果断剪除、果断嫁接

枝干层面的决策需要更频繁的审视。建议以**技术雷达的月度更新 + 06 复盘的 W30/W31 卡点子节**为锚点：

| 动作 | 频率 | 决策标准 |
|------|------|----------|
| **剪除** | 月度 | 连续 4 周没有源码/工程产出的方向，降级为 Hold 或删除 |
| **嫁接** | 季度 | 寻找渲染三特性与 Mac 平台的新交叉点，用一篇追踪建立连接 |
| **加粗** | 持续 | 选择 1 个方向做"单点打穿"——`FVirtualShadowMapCache` 数据流图 + Metal 实现等价物 |

W30 的"单点打穿"目标：**Mac 平台 Anchor 1 实质内容（渲染特性 × Metal 兼容矩阵）**。这个选择是对的：
- Lumen Surface Cache 在 Metal 上的等效实现 → **嫁接 Lumen ↔ Mac 平台**
- Nanite Mesh Pass 的 GPU SW BVH 在 Metal 上的兼容性问题 → **嫁接 Nanite ↔ Mac 平台**
- VSM 页表 LRU 在 Apple Silicon Unified Memory 上的行为差异 → **嫁接 VSM ↔ Mac 平台**

**一个 Anchor 1 同时推动 3 根枝干**，是当前养分利用率最高的动作。

> **枝干法则**：月度审视，季度嫁接，接受剪除的沉没成本。W30 必做：3 份复盘合并决策。

### 树叶的时效性——每周收割，但设定配额

树叶必须存在。没有树叶：
- 不知道这周干了什么
- 复习时没有抓手
- 面试时没有素材
- day-job RAG 索引没法用

但树叶需要**配额管理**。W30 起立硬约束：

| 树叶类型 | 配额 | 理由 |
|---------|------|------|
| QA 卡牌 | **冻结手工生产** | 仅通过 `.tools/gen_*_cards.py` 自动生成；W30 末存量卡牌发出去试推后**只接受反馈迭代，不增新** |
| 复盘日志（06 详细版）| 每周 1 份 | 但聚焦"核心判断 + 承诺兑现" |
| 复盘日志（07 摘要）| 每周 1 份 | 4 段式，与 06 配套 |
| 性能瓶颈案例 | 每周 ≥ 1 条 | W28 0 → W29 3 节奏，W30 续 |
| 雷达 P0 复评 | 每月 1 次 | W28 已做 P0 渲染特性补丁，下次 8/7 |

**关键原则：树叶必须服务于枝干，而不是遮蔽枝干。**

每生产一份树叶，问自己：
1. 这份文档能帮助我下周写出可运行代码吗？（**Mac Anchor 1** → 是；QA 卡牌 → 否）
2. 这份文档能在 day-job RAG 检索时直接命中吗？（**Lumen Surface Cache 数据结构** → 是；QA 卡 "Lumen GI tier 1 是什么" → 是，但已经被卡牌覆盖）
3. 三个月后我还会回看这份文档吗？（**Mac Anchor 1** → 是；W29 QA 卡 → 看 1 次就够）

> **树叶法则**：每周收割，但用"三个月后是否回看"做过滤器。**W30 决策**：QA 卡牌冻结后只做"反馈迭代"，把能量集中到 Mac Anchor 1 + Lumen 升"已掌握"。

## 五、从树到林

单棵树的比喻有局限：它暗示所有养分来自同一根系。GameDevVault 的背景更像一片**异龄混交林**：

- **成熟林层**：Lumen 源码笔记 + Shader C01-C09 + 性能调优 → 胸径粗壮，不需要天天浇水
- **中龄林层**：Nanite / VSM 源码追踪（W29 起头）+ Training Pipeline v2 → 长势良好，需要稳定抚育
- **幼龄林层**：Mac 平台（W29 起头 4.1KB）+ 训练 Pilot 真训 + UE5_Extending_MCP 加 CI → 最需要光照和养分，但目前被成熟林层的落叶遮蔽

**林业的智慧是：不同龄级的林木需要不同的抚育策略。**

| 林层 | 策略 | 具体动作 | W30 必做 |
|------|------|----------|----------|
| **成熟林层**（Lumen 源码 / shader C01-C09 / 性能调优）| 择伐，不是皆伐 | 保持每月 1 篇追踪节奏，不扩展新方向；把数据结构图作为嫁接材料输送给幼龄林 | 把 Lumen Surface Cache 数据结构图**复用**到 Mac Anchor 1 |
| **中龄林层**（Nanite / VSM 源码 / Training v2）| 透光抚育 | W30 续 VSM HZB/CVar + Nanite Cluster/cull/bin/raster + 训练 Pilot 1 epoch | W30 三件套（VSM/Nanite/Pilot）|

## 六、结语：不要问这周产出了多少行，要问这周树干粗了多少

W28 + W29 的共同陷阱是：**用产出量（commits、行数、QA 卡数、文件数）代替生长质量（可运行代码数、引擎源码修改数、Mac 实测次数）。**

W28 30% 兑现率是自爆——5 项 P0 行动只做了 1.5 件。
W29 70% 兑现率是反弹——5 项做了 3 件，剩 2 件需要环境/人。
W30 目标 80%+——剩 2 件必须**真的**做完（Mac Lyra 1 次 + CI 1 个 workflow + Pilot 1 epoch）。

如果 W30 结束时只能记住一件事，应该是这个：

> **树叶（QA 卡 / 复盘 / 论文）的数量是虚荣指标。枝干（源码追踪 / Mac 实测 / 训练模型）的粗度才是成长指标。主干（day-job 方向 = LLM-driven UE on Mac）的稳定是存在指标。**

具体到 GameDevVault 的 W30 行动：
- ❌ **冻结新 QA 卡牌手工生产**（27 → 27，不再涨）

- ✅ **3 份复盘合并决策**（05 Weekly-Log / 06 详细 / 07 摘要 最终分工）

这篇随想本身也是树叶——它会在三个月后泛黄。但如果它能影响下周的养分分配，让 Mac Anchor 1 在 W30 真正填上实质内容、让 Lumen 升"已掌握"的 Mac 跑通门槛有证据、让 QA 卡牌冻结并转向发出去试推——**那它就完成了树叶的使命**：光合作用，然后脱落，把糖分留给枝干。
