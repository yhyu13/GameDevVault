## 被问问题

### 问题 1：

> **问题：** "你提到了 Lumen，那你能详细讲讲 Lumen 的 Radiance Cache 和传统的 Light Probe 有什么区别吗？"

### 我当时的回答

> "Radiance Cache 是 Lumen 特有的，它存储的是辐射度，比 Light Probe 更精确。Light Probe 只是简单的颜色采样，Radiance Cache 用了球谐函数……"

> （**问题：** 这里我混淆了概念。Radiance Cache 确实用球谐，但 Light Probe 也可以用球谐。两者的区别不在于表示方式，而在于**生成方式和更新频率**。）

### 标准/更优答案

> "核心区别在于三点：
> 1. **生成方式**：Light Probe 是预烘焙的，离线生成；Radiance Cache 是实时生成的，每帧根据可见性动态更新。
> 2. **更新频率**：Light Probe 是静态的，除非重新烘焙；Radiance Cache 可以随光源变化实时响应。
> 3. **空间分布**：Light Probe 是规则网格放置；Radiance Cache 是自适应放置，只在需要的地方分配，类似 Sparse Voxel。
> 4. **精度分层**：Lumen 用了两级 Radiance Cache — 远距离 coarse cache 用低阶球谐，近距离 fine cache 用更高分辨率，Light Probe 通常没有这种分层。"

### 问题 2：

> **问题：** "你做过的性能优化案例中，最有挑战的是哪一个？具体怎么解决的？"

### 我当时的回答

> "我做过一个植被 DrawCall 优化，用了 GPU Instancing，从 2000 降到 80 个 DrawCall。"

> （**问题：** 这个回答太浅了。面试官追问 "最有挑战" 时，应该讲一个**有曲折过程**的案例，而不是直接给答案。没有体现出问题分析能力。）

### 标准/更优答案

> "最有挑战的是一个**间接光照的缓存一致性**问题。我们在做 Voxel Cone Tracing 时，发现帧率很不稳定 — 有的角度 60fps，有的角度 30fps。Profile 后发现不是 GPU 瓶颈，而是 CPU 端 voxel 更新时的 cache miss。原因是我们的体素数据结构是 AOS（Array of Struct），每次更新只改一个字段，但把整个结构体都加载到 cache。改成 SOA（Struct of Array）后，cache miss 从 40% 降到 5%，帧率稳定到 60fps。这个案例的难点在于**瓶颈不在预期的地方** — 我一开始以为是 GPU 光追开销，花了两天优化 shader，没效果。后来用 VTune 分析 CPU 才定位到 cache miss。"
