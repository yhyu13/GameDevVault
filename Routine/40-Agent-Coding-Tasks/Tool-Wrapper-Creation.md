# 工具封装创建流程

> **目标**：将常用的 API/服务/功能封装为可复用的工具或脚本
> **触发条件**：发现重复执行同一套操作，或需要向 Agent 暴露新的能力

---

## 封装决策

在封装新工具前，先回答：
1. **必要性**：是否已有工具可覆盖？（避免重复）
2. **通用性**：是否会在 3+ 个不同场景中使用？
3. **稳定性**：依赖的 API/服务是否稳定？
4. **维护成本**：文档、更新、错误处理是否可持续？

---

## 封装流程

### 1. 需求定义
- [ ] 明确工具的输入、输出、副作用
- [ ] 定义错误场景和降级方案
- [ ] 确定调用方式（Python 函数 / Bash 脚本 / MCP 工具）

### 2. 实现
- [ ] 编写核心逻辑
- [ ] 添加输入验证
- [ ] 添加错误处理
- [ ] 添加日志/调试输出
- [ ] 编写使用示例

### 3. 集成
- [ ] 如果是 Python 脚本：放入 `~/.kimi/daimon/scripts/` 或项目目录
- [ ] 如果是 MCP 工具：更新 MCP 配置
- [ ] 如果是 Kimi 原生能力：记录到 `70-References/MCP-Server-Registry.md`

### 4. 文档
- [ ] 编写工具说明（输入、输出、示例、限制）
- [ ] 更新相关技能文档（引用新工具）
- [ ] 添加故障排查指南

### 5. 测试
- [ ] 正常路径测试
- [ ] 边界条件测试
- [ ] 错误路径测试
- [ ] 性能测试（如适用）

---

## 工具质量标准

- [ ] 输入验证：拒绝无效输入，给出清晰错误信息
- [ ] 幂等性：同一输入产生一致输出（或明确标记为有副作用）
- [ ] 可观测：有日志、返回值、状态码
- [ ] 可降级：主路径失败时有替代方案
- [ ] 文档完整：无需阅读源码即可使用
