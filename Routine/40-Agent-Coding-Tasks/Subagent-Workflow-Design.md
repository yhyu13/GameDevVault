# 子智能体工作流设计

> **目标**：设计高效、可维护的子智能体（Subagent）委派模式
> **触发条件**：任务复杂到需要拆分，或需要并行处理多个独立子任务

---

## 设计原则

1. **Context Economy**：子智能体从零上下文开始，只传递必要信息
2. **Single Responsibility**：每个子智能体只负责一个明确的子任务
3. **Timeout 合理**：根据任务复杂度设置超时（默认 1800s，复杂可 3600s）
4. **Resumable**：优先 resume 已有子智能体，而非新建
5. **No Mid-Abandon**：子智能体运行中不要手动接管

---

## 委派决策树

```
任务是否需要并行处理？
├── 是 → 使用多个子智能体，同时发起
│        └── 确保任务独立，无依赖
└── 否 → 任务是否复杂到需要拆分？
         ├── 是 → 使用子智能体顺序执行或传递结果
         │        └── 每个子智能体有明确的输入/输出契约
         └── 否 → 在主 Agent 中直接处理
                  └── 避免不必要的 context handoff 成本
```

---

## 工作流模式

### 模式1：Map-Reduce（分治）
- **场景**：大量独立项需要相同处理（如批量分析文件）
- **实现**：多个子智能体并行处理 → 主 Agent 汇总结果
- **注意**：确保子智能体输入不重复，结果格式统一

### 模式2：Pipeline（流水线）
- **场景**：多阶段任务，每阶段输出是下一阶段输入
- **实现**：子智能体 A → 结果 → 子智能体 B → 结果 → 子智能体 C
- **注意**：中间结果格式契约，错误传播机制

### 模式3：Expert Panel（专家小组）
- **场景**：需要多角度分析同一问题
- **实现**：多个子智能体从不同视角分析 → 主 Agent 综合判断
- **注意**：视角定义清晰，避免重复分析

### 模式4：Verification（验证）
- **场景**：关键任务需要复核
- **实现**：子智能体执行 → 另一子智能体验证 → 主 Agent 仲裁
- **注意**：验证标准明确，仲裁规则清晰

## 设计检查清单

- [ ] 子任务边界清晰？
- [ ] 输入/输出契约明确？
- [ ] 超时设置合理？
- [ ] 错误处理路径明确？（子智能体失败怎么办？）
- [ ] 结果合并策略清晰？
- [ ] 是否优先 resume？

---

## 常见陷阱

1. **过度拆分**：任务太简单，子智能体 overhead > 收益
2. **Context 爆炸**：传递过多上下文，导致每个子智能体都重复理解
3. **循环依赖**：子智能体 A 依赖 B，B 依赖 A
4. **无超时**：默认 30min 可能不够，也可能太长
5. **过早合并**：结果还没完全产生就合并，导致数据缺失
