# UE5 训练数据生成指南

> **核心理念**：你是领域专家，AI 是格式转换器。你写内容，AI 帮你变成标准 `.jsonl`。数据质量 > 数据数量。

## 为什么自动提取不够好？

| 自动提取的问题 | 人类生成的优势 |
|-------------|-------------|
| 把 markdown 直接塞给模型当 output，太长太冗余 | 你可以提炼核心要点，写出精准的回答 |
| 追问链没有真实答案，只有占位符 | 你知道面试官真正想听到什么 |
| 代码解释只是复制上下文，没有深度 | 你能讲清楚 "为什么这段代码要这样写" |
| 缺乏错误示范（模型只学对的不学错的） | 你可以写 "常见错误 + 纠正" 对 |
| 没有跨模块关联（Nanite 和 Lumen 怎么配合） | 你有项目经验，知道真实协同关系 |

## 高质量数据的 5 个标准

### 1. Instruction 要具体
❌ 坏例子：`"解释 Nanite"`  
✅ 好例子：`"面试官问：Nanite 为什么要用软件光栅化处理亚像素三角形？硬件光栅化有什么问题？"`

### 2. Output 要有结构
❌ 坏例子：一段大文字没有层次  
✅ 好例子：
```
1. 硬件光栅化的局限：
   - 三角形小于 1 像素时产生 Quad Overdraw（2×2 像素块只覆盖 1 个有效像素）
   - Early-Z 失效，Overdraw 增加
2. 软件光栅化（Compute Shader）的优势：
   - 逐像素精确计算覆盖，没有 Quad Overdraw
   - 适合处理微多边形的亚像素三角形
3. 源码位置：Engine\Source\Runtime\Renderer\Private\Nanite\NaniteRasterizer.cpp
```

### 3. 覆盖常见错误（Hard Negative）
模型不仅要知道什么是对的，还要知道**为什么某些看起来很对的说法是错的**：

```json
{
  "instruction": "面试官说：'Nanite 的 GPU-Driven 意味着 CPU 完全不参与渲染。' 这个说法对吗？",
  "output": "不对。CPU 仍参与离线构建阶段（Cook 时生成 Cluster/DAG/Page），运行时才是 GPU-Driven。混淆了 '运行时' 和 '完整流程'。"
}
```

### 4. 包含追问链（多轮对话）
真实面试不是一问一答，而是层层深入。写出完整的追问链：

```
Q1: 什么是 Surface Cache？
A1: ...（简要回答）

Q2（追问）：如果 Card 太多导致更新不及时，会有什么现象？
A2: ...（深入回答）

Q3（再追问）：CardCapturesPerFrame 调大能完全解决这个问题吗？
A3: ...（带权衡的回答）
```

### 5. 代码要有上下文
不要只给代码，要给 **"这段代码在干什么 + 为什么这样写 + 如果改会怎样"**：

```json
{
  "instruction": "分析这段 UE5 源码的作用：void FStreamingManager::RequestPages(const FViewInfo& View)",
  "output": "这是 Nanite 的流式加载入口。根据当前视锥和 HZB 计算可见 Cluster，只加载对应 Page。如果漏掉 HZB 检查，会导致大量被遮挡的 Page 也被加载，显存爆炸。"
}
```

## 数据类型模板

本目录提供 4 种数据类型的模板，你在 `human_data/` 目录下按模板写 markdown，我帮你转成 `.jsonl`。

| 模板文件 | 用途 | 适合写什么 |
|---------|------|----------|
| `template_qa.md` | 单轮问答 | 概念解释、源码分析、技术对比 |
| `template_code.md` | 代码解释 | 源码片段 + 深度分析 |
| `template_conversation.md` | 多轮追问 | 面试模拟、层层深入的对话 |
| `template_negative.md` | 错误纠正 | 常见误区 + 纠正 |

## 工作流程

```
Step 1: 你选一个模板，在 human_data/ 下创建 .md 文件
        ↓
Step 2: 按模板格式写内容（用你的专业知识）
        ↓
Step 3: 运行 python scripts/convert_human_data.py
        ↓
Step 4: 检查生成的 .jsonl，修正不满意的地方
        ↓
Step 5: 重复 Step 1-4，直到覆盖所有你想让模型学会的知识点
```

## 建议的数据量

| 知识点领域 | 建议 QA 对 | 建议对话链 | 建议代码对 |
|-----------|----------|----------|----------|
| Nanite | 20-30 | 5-8 | 10 |
| Lumen | 20-30 | 5-8 | 10 |
| VSM | 15-20 | 3-5 | 5 |
| GPU-Driven Pipeline | 10-15 | 3-5 | 5 |
| Render Graph | 10-15 | 2-3 | 5 |
| **总计** | **~100** | **~25** | **~35** |

> 100 条高质量人工数据 > 1000 条自动提取数据。

## 数据 Review Checklist

写完一批数据后，逐条检查：

- [ ] Instruction 是否像真实面试问题？
- [ ] Output 是否包含源码路径或具体函数名？
- [ ] 是否有 "为什么" 的解释，而不只是 "是什么"？
- [ ] 是否有至少一条包含常见错误纠正？
- [ ] 多轮对话的追问是否有逻辑递进？
- [ ] 代码解释是否说明了 "如果改会怎样"？
