## 1. The LLM Training Pipeline Overview

Every modern LLM is built through a **staged, sequential pipeline** — not a single "press train" operation. The 2026 frontier-lab pipeline has evolved from the simple two-stage recipe (pre-training → RLHF) into a modular four-stage architecture [^1][^2].

```
Pre-training (10-15T tokens, language foundation)
    ↓
Mid-training / Continual Pre-training (domain specialization, optional)
    ↓
SFT — Supervised Fine-Tuning (5K-500K examples, format & instruction)
    ↓
Preference Optimization — DPO / IPO / KTO (10K-100K pairs, alignment)
    ↓
RL with Rewards — GRPO / PPO (reasoning & safety refinement)
    ↓
RLVR — RL with Verifiable Rewards (math, code, structured tasks)
    ↓
Final SFT / Distillation (consolidation)
```

**Pre-training** teaches the model to predict the next token across trillions of tokens. It does not produce an assistant — it produces a text-completer. **Post-training** transforms that completer into a useful, safe, reasoning-capable assistant [^2].

In 2025-2026, the dominant narrative shifted: **post-training accounts for the majority of usable model capability**. Liquid AI benchmarks show post-training alone improves performance by 20-40% across standard evaluations — an improvement that would require orders of magnitude more pre-training compute to achieve through scaling alone [^3]. The old "RLHF recipe" (SFT → PPO) has been displaced by a modular stack: SFT → DPO → GRPO/RLVR [^4].

## 2. Where Human Experts Fit (SFT, RLHF, Curation, Evaluation)

Human domain experts are not optional luxuries — they are **bottleneck resources** at multiple stages. The quality of their judgment directly determines the quality of the final model.

### Stage 1: SFT — Demonstration Data Writers
Experts write prompt-response pairs that teach the model *what* to say and *how* to say it. OpenAI's InstructGPT used 40 highly-educated human labellers who passed a screen test to produce demonstration data [^5]. For domain-specific models (e.g., code, medicine, game development), generic annotators are insufficient — you need practitioners who can write correct, idiomatic, expert-level responses [^6].

### Stage 2: Preference Alignment — Rankers & Judges
Experts rank or score model-generated outputs. This creates the training signal for DPO/RLHF. The work is harder than it sounds: experts must evaluate outputs across multiple dimensions (accuracy, helpfulness, safety, clarity) and provide consistent rankings [^7].

### Stage 3: Data Curation & Quality Filtering
Experts define what "quality" means for their domain. A game developer expert decides whether a piece of UE5 Blueprint code or a rendering pipeline explanation is pedagogically sound, syntactically correct, and follows industry best practices.

### Stage 4: Evaluation & Red-Teaming
Experts construct adversarial prompts, verify correctness of model outputs, and identify failure modes. This is where a UE5 expert would test whether the model correctly understands Niagara particle systems, Lumen GI, or the difference between static and skeletal mesh workflows.

### Stage 5: Synthetic Data Validation
In the emerging "AI-first, human-reviewed" paradigm, experts do not write every example from scratch. Instead, they review, correct, and validate synthetic data generated by larger models (see Section 6).

**Compensation:** Expert annotators command premium rates. RLHF preference annotator roles are advertised at $60-85/hour [^8], and PhD-level domain experts for hard reasoning tasks (STEM, law, medicine) are reportedly paid $100-200/hour at frontier labs.

## 3. Demonstration Data: What Experts Actually Write

### What Is a "Golden Example"?

A demonstration data point for SFT is a **prompt-response pair** that demonstrates the exact behavior the model should learn. The "golden example" format varies by task but typically follows an instruction-following structure [^5][^9]:

```json
{
  "instruction": "Write a Python function that implements A* pathfinding for a 2D grid.",
  "input": "Grid: 10x10, start=(0,0), goal=(9,9), obstacles=[(3,3),(3,4),(3,5)]",
  "output": "Here is a clean, well-documented A* implementation...\n\n```python\ndef astar(grid, start, goal):\n    ...\n```\n\n**Explanation:** The algorithm uses a priority queue (heapq) to explore nodes by f-cost. The heuristic is Euclidean distance..."
}
```

### Key Properties of High-Quality Demonstration Data

| Property | Why It Matters |
|----------|---------------|
| **Correctness** | The response must be factually and technically correct. One wrong example teaches the model to hallucinate. |
| **Completeness** | The response should address all parts of the prompt, not just the easiest. |
| **Clarity** | Step-by-step reasoning, clear structure, and appropriate formatting. |
| **Consistency** | All examples should follow the same style, tone, and depth conventions. |
| **Diversity** | Cover edge cases, common mistakes, and varying difficulty levels. |
| **Idiomaticity** | For code, use the target language's conventions. For UE5, use engine-specific terminology (e.g., "Actor," "Component," "Blueprint"). |

### How Many Examples Are Needed?

The quantity requirements have dropped dramatically. Modern practice suggests:
- **SFT:** 5,000 to 50,000 high-quality prompt-response pairs for general alignment [^4]
- **Domain-specific SFT:** Sometimes as few as 1,000-3,000 expertly curated examples (the LIMA result showed GPT-4 preference parity with only 1,000 curated examples) [^10]
- **Code SFT:** Phi-1 was fine-tuned on only ~180M tokens of synthetic exercises (roughly equivalent to thousands of exercises) and achieved 50.6% on HumanEval [^11]

**The rule:** Quality trumps quantity by an order of magnitude. A smaller, cleaner dataset curated by domain experts outperforms a massive noisy dataset [^10][^11].

### Concrete Example: Code SFT Data

A good code SFT example includes:
1. **Problem statement** (natural language)
2. **Function signature** (type hints, docstring)
3. **Solution** (clean, idiomatic, commented code)
4. **Test cases** (input/output pairs for verification)
5. **Explanation** (why the solution works, complexity analysis)

For UE5/game dev specific training, the equivalent would be:
1. **Task description** (e.g., "Implement a custom Actor that spawns particles on collision")
2. **Context** (engine version, relevant classes, expected behavior)
3. **Solution** (C++ or Blueprint code, following UE5 conventions)
4. **Rationale** (why this approach, alternatives considered, performance implications)

## 4. Preference Data: How Experts Rank Outputs

### What Is Preference Data?

Preference data consists of pairs (or groups) of model-generated responses to the same prompt, with human judgments indicating which is better. This is the fuel for DPO, RLHF, and KTO [^12][^13].

### The Pairwise Comparison Format

```json
{
  "prompt": "Explain how Lumen global illumination works in Unreal Engine 5.",
  "chosen": "Lumen is UE5's dynamic global illumination system. It works by... [correct, detailed, well-structured explanation]",
  "rejected": "Lumen is a lighting thing in UE5. It makes lights look better by bouncing them around. [vague, technically imprecise]"
}
```

### Expert Ranking Rubric: Multi-Axis Scoring

Frontier labs do not ask annotators for a single "better/worse" judgment. They use structured rubrics [^7][^14]:

| Dimension | Description | Weight |
|-----------|-------------|--------|
| **Accuracy** | Factually correct? Technical details precise? | High |
| **Helpfulness** | Does it actually solve the user's problem? | High |
| **Clarity** | Well-structured? Easy to follow? | Medium |
| **Completeness** | Addresses all parts of the prompt? | Medium |
| **Safety** | No harmful, biased, or illegal content? | Critical (hard filter) |
| **Style** | Appropriate tone and formatting? | Low |

A domain expert (UE5) would add domain-specific axes:
- **Engine-version correctness** (does it reference UE5 features, not UE4?)
- **API correctness** (are the class names, function signatures, and Blueprint node names correct?)
- **Performance awareness** (does the solution consider frame budget, draw calls, memory?)
- **Best-practice alignment** (does it follow Epic's recommended patterns?)

### Inter-Annotator Agreement (IAA)

Consistency between experts is critical. Teams track **Cohen's kappa** or **Krippendorff's alpha**, with targets of κ ≥ 0.7 (substantial agreement) or κ ≥ 0.85 (near-perfect) for high-stakes domains [^6][^14]. If three experts rank the same set of responses and their rankings do not correlate, the instructions are unclear or the task requires deeper expertise than the annotators possess [^6].

**Consensus mechanisms:** Critical samples are assigned to 3-5 annotators. The final label is determined by majority vote. Honeypot tasks (known-good and known-bad examples) are inserted to catch systematic bias [^14].

### RLTHF: The 6-7% Human Annotation Paradigm

A recent breakthrough (ICML 2025) shows that full-human-annotation-level alignment can be achieved with only **6-7% of the human annotation effort** [^15]. RLTHF (Reinforcement Learning from Targeted Human Feedback) works by:
1. Using an LLM to label the full dataset initially
2. Training a reward model on the LLM-labeled data
3. Using the reward model's distribution to identify hard-to-annotate samples (where the LLM is likely wrong)
4. Sending only those hard samples to human experts for correction
5. Iterating

This is the emerging paradigm for expert involvement: **experts are not replaced — they are targeted at the exact problems where AI fails** [^15].

## 5. Data Curation & Quality Filtering (Domain-Specific Considerations)

### The Curation Pipeline

Raw data is not training data. The transformation pipeline includes [^16][^17]:

1. **Collection** — scraping, API ingestion, synthetic generation
2. **Deduplication** — exact + near-duplicate removal (MinHash, SemDeDup)
3. **Quality Filtering** — classifier-based or heuristic-based
4. **Privacy Reduction** — PII removal
5. **Domain Mixing** — balancing data sources (DoReMi, RegMix)
6. **Expert Validation** — human review of edge cases

### DoReMi: Optimizing Data Mixtures

DoReMi trains a small proxy model using Group DRO to find optimal domain mixture weights, then uses those weights to train the full model. Results show **2.6x speedup with 6.5% better downstream accuracy** [^18]. For a UE5 expert, this means determining the optimal ratio of:
- UE5 official documentation
- Community tutorials (Unreal Engine forums, YouTube transcripts)
- C++ code samples
- Blueprint visual-scripting examples
- Game design pattern discussions
- Related domains (graphics, physics, networking)

### How Domain Experts Define "Quality"

A generic quality filter (perplexity score, language classifier) cannot evaluate whether a UE5 tutorial correctly explains the difference between `Tick()` and `BlueprintEvent` usage. Domain experts define quality through:

- **Technical correctness** — Does the code compile? Does the explanation match engine behavior?
- **Pedagogical soundness** — Is it teachable? Does it build on prerequisites?
- **Idiomaticity** — Does it follow Epic's style guidelines and community conventions?
- **Recency** — Does it reference current engine APIs (UE5.4 vs. UE4.27)?
- **Completeness** — Does it cover common failure modes?

The rule of thumb from the literature: **coarse-to-fine pipelines win**. Cheap rule-based filters remove the obviously bad; model-based operators concentrate compute on the valuable remainder; expert validation catches the subtle errors that automated filters miss [^17].

## 6. Synthetic Data + Expert Validation (The Hybrid Workflow)

### The Phi-1 Breakthrough

Microsoft's Phi-1 (2023) proved that **high-quality synthetic data can break scaling laws**. A 1.3B-parameter model trained on only 7B tokens (including 1B synthetic "textbook-quality" Python textbooks and 180M tokens of synthetic exercises) achieved 50.6% on HumanEval — competitive with models 10x larger [^11][^19].

### The "Textbook Quality" Standard

The Phi series data was generated by GPT-3.5 and then filtered/validated. Key principles [^11][^20]:
- **Diversity:** Cover a wide range of concepts, skills, and scenarios
- **Difficulty gradation:** Varying complexity from beginner to advanced
- **Correctness:** All exercises have verified solutions
- **Non-repetition:** Minimal duplication to avoid memorization
- **Expert review:** Human validation of synthetic outputs

### The Hybrid Workflow: AI-First, Human-Reviewed

The modern synthetic data pipeline works like this [^20][^21]:

```
1. Seed → Human experts write a small set of high-quality seed examples
2. Generate → Large model (GPT-4, Claude, DeepSeek) generates synthetic variations
3. Filter → Automatic filters remove duplicates, low-quality, and off-topic outputs
4. Verify → Execution-based verifiers (for code) or rule-based checkers validate correctness
5. Expert Review → Domain experts spot-check, correct, and approve the final dataset
6. Iterate → The curated dataset becomes the seed for the next generation round
```

### Synthetic Data Risks: Model Collapse

Pure synthetic data without real data retention leads to **model collapse** — perplexity degrades by 20-28 points after five epochs of pure synthetic self-training [^20]. The fix is simple: **accumulate real data alongside synthetic data rather than replacing it**. Optimal mix: 33-67% synthetic + the rest real data, with at least 10% real data retained to limit drift [^20].

### Cosmopedia: The Open Replication

Hugging Face's Cosmopedia replicated the Phi approach openly: 25B synthetic tokens from 30M+ prompts, with <1% duplicates. This validates that the synthetic approach is reproducible outside of Microsoft Research [^20].

## 7. Reasoning Models: New Paradigms (DeepSeek-R1, o1 Style Training)

### The Shift: From Preference Learning to Verifiable Rewards

Traditional alignment (RLHF/DPO) teaches the model *what humans prefer*. Reasoning training (DeepSeek-R1, OpenAI o1/o3) teaches the model *how to think step-by-step correctly* — using **verifiable rewards** rather than human preferences [^22][^23].

### DeepSeek-R1's Four-Stage Pipeline

1. **Cold Start:** Collect thousands of high-quality chain-of-thought (CoT) reasoning examples. These are curated by human experts or distilled from strong teacher models. Unlike DeepSeek-R1-Zero (which used pure RL with no SFT), R1 uses a small but high-quality SFT warm-start to ensure readable, well-formatted reasoning [^22][^24].

2. **Reasoning-Oriented RL:** Apply GRPO with rule-based rewards. The reward function is simple: **R = R_accuracy + λ * R_format**. For math: does the final answer match? For code: do the unit tests pass? [^22][^25].

3. **Rejection Sampling + SFT:** Near RL convergence, generate a large synthetic dataset by sampling from the trained model and keeping only correct, high-quality trajectories. Merge with general-domain data (writing, factual QA) and do another SFT round [^23][^24].

4. **Final RL:** A second RL stage across all scenarios, balancing rule-based rewards for reasoning with reward-model-based rewards for helpfulness and safety [^22].

### Cold Start Data: Where Experts Are Critical

The "cold start" data for reasoning models is **small but precious**. DeepSeek-R1 used only "thousands" of examples — but each one was a long, carefully structured CoT with reflection, verification, and final summary [^22]. A UE5 expert's role here would be to write reasoning traces like:

```
Problem: "Why are my Lumen reflections flickering in this scene?"

Reasoning:
1. Lumen reflections depend on screen-space data and ray-traced data.
2. Flickering suggests temporal instability — the reflection data is changing frame-to-frame.
3. Check: Is the Surface Cache resolution too low? Low res causes popping.
4. Check: Are there emissive materials without proper Lumen importance volumes?
5. Check: Is the project using Software Ray Tracing on a complex mesh? Switch to Hardware RT if available.

Verification: The most common cause is Surface Cache resolution. Try `r.Lumen.ScreenProbeGather.RadianceCache.Resolution = 8` or higher.

Summary: Flickering Lumen reflections are usually caused by insufficient Surface Cache resolution or missing importance volumes on emissive surfaces.
```

### Process Rewards vs. Outcome Rewards

| Type | Definition | Use Case | Expert Role |
|------|-----------|----------|-------------|
| **Outcome Reward (ORM)** | Binary signal: final answer correct? | Math, code with unit tests | Define correctness criteria |
| **Process Reward (PRM)** | Score each intermediate reasoning step | Long-chain reasoning where one small error ruins the answer | Define step-level rubrics, validate step annotations |
| **Verifiable Process Reward (VPRM)** | Deterministic rule-based check at each step | Structured reasoning (medical diagnosis, legal analysis, technical debugging) | Write the domain-specific verification rules |

Process rewards are powerful but expensive: OpenAI's PRM800K required 800,000 human-annotated step-wise labels [^26]. Recent work (VPRM, ProcessThinker) replaces neural process judges with deterministic verifiers, but **someone still has to write the domain rules** — that is the expert's role [^27][^28].

## 8. Code Training Specifics (Execution Feedback, Test Cases)

### Why Code Is Different

Code is the ideal domain for LLM training because it has **objective, automatic verification**: a program either compiles and passes tests, or it does not. This enables **RLVR** (Reinforcement Learning with Verifiable Rewards) — the most significant paradigm shift in 2025-2026 post-training [^29][^30].

### The Code Training Pipeline

1. **Pre-training on code:** 2T+ tokens of code and natural language (e.g., DeepSeek-Coder, CodeLlama) [^31]
2. **SFT on instruction-code pairs:** Expert-written or synthetic problem-solution-explanation triplets [^31]
3. **RLVR with execution feedback:** Generate code → run unit tests → binary reward (pass/fail) → update model [^29]
4. **Self-debugging training:** Models learn to read execution error messages, explain bugs, and generate fixes [^32]

### Execution Feedback as Reward

The reward function for code RL is brutally simple and effective [^29][^30]:

```python
def reward(code, test_cases):
    for test in test_cases:
        if not run(code, test.input) == test.expected:
            return 0  # fail
    return 1  # pass
```

This is **verifiable** — no human preference needed. It is also **sparse** — the model gets zero signal until the entire solution is correct. Recent work (StepCoder, RLTF) addresses sparsity by providing fine-grained rewards through error locations from unit test feedback [^31].

### CodeScaler: Execution-Free Reward Models

Not all code problems have perfect test cases. CodeScaler trains a learned reward model to predict code quality without execution, using syntax-aware code extraction and validity-preserving reward shaping to mitigate reward hacking [^29]. This is the bridge between pure RLVR (which needs verifiable tests) and domains where verification is expensive or incomplete.

### Expert Role in Code Training

For a UE5/code expert, the role is to:
- **Write test cases:** Define the correctness criteria. A solution is only as good as its test coverage.
- **Design problem sets:** Ensure diversity of difficulty, concepts, and engine subsystems.
- **Validate synthetic solutions:** Review AI-generated code for correctness, idiomaticity, and performance.
- **Define execution environments:** Set up the sandbox (e.g., UE5 editor automation, compilation pipeline) that provides the RL signal.

## 9. Tools & Platforms (What Interfaces Experts Use)

### Three Categories of Annotation Tools

| Category | Examples | Best For |
|----------|----------|----------|
| **Labeler Marketplaces** | Scale AI, Surge AI, Appen | High-volume, managed workforce, pre-training grade |
| **Self-Service Platforms** | Labelbox, Argilla, Label Studio, Encord | Teams who own annotation as a discipline |
| **In-Product Queues** | Braintrust, Future AGI | Tight loop with traces, evals, and judges |

### Scale AI & Surge AI
Scale AI is the largest managed workforce for LLM annotation, offering RLHF preference-ranking projects with domain-specialist recruitment. Surge AI specializes in calibrated RLHF and instruction-tuning lineages [^33]. Both operate on quote-based pricing for enterprise volumes.

### Label Studio / Argilla (Open Source)
Label Studio (HumanSignal) and Argilla (Hugging Face) are Apache 2.0 tools that support pairwise ranking, preference templates, and inter-annotator agreement tracking [^34]. They are the go-to for teams building internal annotation factories.

### Key Features Experts Need

| Feature | Why It Matters |
|---------|---------------|
| **Pairwise comparison interface** | Side-by-side model outputs for ranking |
| **Multi-axis rubric scoring** | Rate accuracy, helpfulness, safety independently |
| **Consensus workflows** | Route critical tasks to multiple experts |
| **IAA tracking** | Measure Cohen's kappa or Krippendorff's alpha |
| **Honeypot/sentinel tasks** | Catch systematic bias with known-answer checks |
| **Real-time feedback loop** | Guidelines evolve as edge cases are discovered |
| **Model-assisted pre-labeling** | LLM generates first draft, human edits (40-60% speedup) [^35] |

### What an Expert's Day Looks Like

A typical domain expert annotator's workflow [^8][^14]:
- **Morning:** Review updated annotation guidelines (rubrics change weekly based on model drift)
- **Work blocks:** 3-6 hour focused sessions evaluating prompt-response pairs against rubrics
- **Calibration:** Weekly team meeting to align on tricky edge cases and updated policies
- **Volume:** 20-30 hours/week, with targets for throughput and agreement rates
- **Compensation:** $60-200/hour depending on domain depth and expertise level

## 10. Key Takeaways for UE5 Expert

### What "AI Training Direction Game Development Expert" Actually Means

You are not being hired to write code for the model. You are being hired to **steer the model's behavior in the game development domain** through structured data creation and judgment. Your job is to act as a **human compass** for the AI.

### Your Primary Responsibilities

1. **Write golden-example SFT data:** Create prompt-response pairs that demonstrate expert-level UE5 knowledge (Blueprint, C++, rendering, physics, Niagara, etc.)
2. **Rank model outputs:** Evaluate which of two generated UE5 explanations is more accurate, helpful, and idiomatic
3. **Define domain quality:** Decide what makes a UE5 training example "textbook quality" vs. "garbage"
4. **Validate synthetic data:** Review AI-generated UE5 tutorials, code snippets, and explanations for correctness
5. **Design test cases:** For code-training models, write unit tests that verify UE5 API usage correctness
6. **Curate domain mixtures:** Determine the optimal ratio of UE5 docs, tutorials, code, and related graphics/engineering content

### Why This Role Is Valuable

- **Generic annotators cannot do your job:** A crowd worker does not know the difference between `UPROPERTY` specifiers or why `Tick()` frequency matters for frame budgets. That knowledge is your moat.
- **You are the filter:** In an AI-first world, your role shifts from "content creator" to "quality gatekeeper." The AI generates; you validate, correct, and curate.
- **The 6-7% principle:** You do not need to annotate everything. RLTHF shows that targeted expert feedback on the *hardest* cases achieves full alignment with minimal effort [^15].
- **The shift to verifiable rewards:** For code and structured reasoning, the model's reward comes from unit tests and rule-based verifiers — not human preferences. Your job is to **write those rules and tests** [^29][^30].

### Concrete Skills to Develop

| Skill | Application |
|-------|-------------|
| **Structured writing** | Formatting prompt-response pairs for SFT |
| **Rubric-based evaluation** | Consistent ranking of model outputs |
| **Test-case design** | Unit tests for RLVR code training |
| **Data quality assessment** | Spotting errors in synthetic UE5 content |
| **Domain taxonomy design** | Defining the skill graph for game dev AI |

### The Bottom Line

The LLM training pipeline in 2026 is a **modular, human-AI hybrid system**. Your role as a UE5 expert is not to compete with the AI but to **channel your domain expertise into the exact places where AI fails**: writing the first high-quality examples, ranking ambiguous outputs, validating synthetic data, and defining the verifiable correctness criteria that turn a generic language model into a competent game development assistant.
